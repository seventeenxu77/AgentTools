description: Bug 修复复盘报告生成

复盘**当前会话**刚修复的 bug，产出一份结构化复盘报告。不使用 Plan subagent，严格按顺序执行。

可选聚焦提示：$ARGUMENTS（为空则复盘本次会话的主 bug；非空则聚焦指定的那个 bug/模块）。

## 核心守卫：只复盘，不编造

- 报告内容**只来自当前会话真实发生的修复过程**——你做过的调查、读过的代码、改过的方法。
- 会话上下文里没有的信息（尤其第 2 节「调查思路链路」），**不许臆造填充**。缺什么就如实标「未记录 / 需用户补充」，或停下来问用户，绝不为凑满模板编造过程。
- 每个结论都要给**依据**：`文件:行`、方法名、配置表名、事件名。无依据的判断显式标注「推测」。

## 撰写风格

以**数据 / 调用链为中心**，用箭头串联（`A.方法 → B字段 → C系统`），根因导向，少贴大段代码（只引关键行）。建议对齐你项目 commit message 的写法：先点根因，再用 `→` 展开链路。

## 步骤

1. 在当前会话上下文中锚定本次要复盘的 bug（参考 `$ARGUMENTS`）。若会话里根本没有可复盘的修复过程，直接告知用户并停止，不要凭空生成。
2. 将报告写入 `.temp/docs/Bug复盘_<bug一句话名>.md`，**中文命名**。报告含以下 5 节：

### 1. 本次修复了什么 bug
- **现象**：用户视角看到的错误表现
- **影响范围**：波及哪些功能 / 系统
- **一句话根因**

### 2. 为什么出现 + 如何定位（调查思路链路）
- **根本原因**：机制层面的成因（不是表象）
- **调查链路**：本次会话**真实**的定位过程，按 `线索 → 怀疑 → 验证 / 排除 → 收敛` 记录
- **关键转折**：哪个发现让定位收敛到根因

### 3. 涉及代码位置 + 如何传播到其他系统
- **代码位置**：`文件:行` + 关键片段（只引核心几行）
- **传播链**：这段代码通过哪些方法 / 事件 / 数据把错误扩散出去，以数据为中心画链路：`出错点 → 中间方法 / 字段 → 受害系统`

### 4. 修复的关键方法 + 作用链路
- 逐个列出修复改动的**关键方法**（`文件:方法`）
- 每个：改了什么 + 它在调用链中的位置 + 修复后链路如何正确运转
- **修复前 vs 修复后**链路对比

### 5. 验证 & 防复发
- **验证**：如何确认修复有效（复现步骤 / 编译命令 / 运行观察点）
- **防复发**：是否需补参考文档、加防御性断言 / 检查、补测试；同类隐患是否在别处也潜伏

3. 输出文件路径告知用户，等待确认或补充。若用户指出某节信息有误或不全，据实修订，不强行自圆其说。
