---
description: Bug 定位与修复流程
---

不使用 Plan subagent，严格按照顺序执行：

## 1. 前置：检查 Git 改动

在查文档/读代码之前，先排查 bug 是否由近期改动引入：
- 执行 `git status` 查看未提交的本地改动
- 执行 `git log --oneline -10` 查看最近提交
- 若发现与 bug 现象可能关联的文件/模块（同系统、同链路、同数据），使用 `git diff <file>` 或 `git show <commit>` 查看具体变更
- 将这些改动作为**优先怀疑点**纳入后续分析；若直接命中根因，可跳过文档查阅与对比步骤，直接进入步骤 5

## 2. 定位：查阅直接相关文档

检索项目的参考/设计文档（用你项目的文档索引工具；若没有，直接用 Grep/Glob 搜 docs 目录）。
基于文档标题/摘要筛选与 bug 直接涉及的系统文档并阅读。

## 3. 扩展：查阅潜在关联文档

基于步骤 2 的理解，扩展检索可能关联的系统文档或设计文档（上下游依赖、共享数据、事件链路）。

## 4. 对比：参考相似系统的实现

使用 Explore Agent 搜索相似系统或设计的代码。
输出：
- 参考系统与当前系统的**关键差异**
- 可复用的代码或设计模式
- 当前实现与参考实现的偏差点（可能即为 bug 根因）

## 5. 陈述：修改方案确认

在动手修改前，先向用户说明：
- **修改位置**：具体文件与函数
- **修改原因**：为什么要在这里改，与 bug 根因的关系
- **关键变更**：代码层面的核心修改内容，展示给用户看

等待用户确认后再执行。

## 6. 处理：修复或插桩调试

- **简单问题**：向用户说明根因和修复方案，经用户同意后直接修复。
- **复杂问题**：向用户说明当前分析结论和不确定点，建议在关键函数或系统节点插入 debug 日志。
  - debug 日志格式：`[bug-{关键词}]`，关键词需具有**唯一可搜索性**（如 `[bug-power-consume]`）。
  - 列出建议插桩位置，等待用户确认后执行。

## 7. 清理：debug 标签

问题解决后，提示用户是否清除本次插入的 `[bug-*]` 标签。
经用户同意后，使用 Grep 搜索对应标签并清除。

## 8. 清理：debug 临时文档

如本次会话中生成了调试相关的临时文档，经用户同意后删除。

## 9. 更新：参考文档

如果本次修复揭示了文档中的缺失或错误，更新相关参考文档。

## 10. 提交：相关代码

询问用户是否 git 提交代码
